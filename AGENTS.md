# Инструкции для репозитория

## Структура проекта и организация модулей

Это **база знаний, поддерживаемая LLM**, по паттерну LLM Wiki (Karpathy, 2024). Репозиторий не содержит системы сборки, тестов или пайплайна деплоя — это структурированная коллекция Markdown-файлов.

```
.
├── .schema/               # Правила и соглашения рабочего процесса (только чтение)
│   └── CLAUDE.md          # Полная схема для LLM-агентов
├── raw/                   # Неизменяемые исходные документы (LLM никогда не редактирует)
│   ├── articles/          # Статьи, PDF, веб-клиппинг
│   └── assets/            # Изображения и медиа
└── wiki/                  # Вики-страницы, сгенерированные LLM
    ├── index.md           # Каталог всех страниц (обновляется при каждом изменении)
    ├── log.md             # Журнал операций (только дополнение)
    ├── concepts/          # Страницы концепций и тем
    ├── entities/          # Люди, организации, проекты
    ├── sources/           # Сводки источников
    └── queries/           # Сохранённые ответы на запросы
```

## Стиль кода и соглашения об именовании

- **Заголовки страниц и названия концепций**: английский (`react-state`, `aspnetcore-basics`).
- **Тело страниц**: русский (родной язык пользователя). Код и технические термины остаются на английском.
- **Имена файлов**: lowercase kebab-case (напр. `django-authentication.md`).
- **Перекрёстные ссылки**: Obsidian `[[wikilinks]]` — использовать активно.
- **Структура страницы**: каждая вики-страница должна содержать YAML frontmatter:

```yaml
---
title: "Название страницы"
tags: [тег1, тег2]
created: ГГГГ-ММ-ДД
updated: ГГГГ-ММ-ДД
sources: [source-file.md]
---
```

## Команды сборки, тестирования и разработки

В этом репозитории **нет системы сборки, тестового фреймворка или dev-сервера**. Вся работа ведётся прямым редактированием Markdown-файлов. Доступные команды:

| Команда | Назначение |
|---------|------------|
| `python -m py_compile <файл>` | Валидация синтаксиса YAML frontmatter (опционально) |
| Нет тестового раннера | Проверка корректности через чтение страниц и проверку `[[wikilinks]]` |

Файлы в `raw/` никогда не изменяются программно — они ингестируются как есть.

## Правила тестирования

Автоматизированное тестирование к этому проекту не применяется. Целостность данных поддерживается через:

- **Периодический lint**: проверка битых `[[wikilinks]]`, страниц-сирот, противоречий и устаревших утверждений.
- **Ручную проверку**: перед ответом на запрос читать релевантные вики-страницы; сверять, что `wiki/index.md` покрывает все существующие страницы.
- **Дисциплина index**: `wiki/index.md` всегда должен отражать реальное состояние всех страниц.

## Правила коммитов и пул-реквестов

Это однопользовательский репозиторий, управляемый LLM. При коммитах:

- Используйте описательные сообщения в настоящем времени, указывающие операцию и область:
  - `ingest: process htmx-examples source`
  - `lint: fix broken wikilinks and orphan pages`
  - `update: add security section to django-views`
- Pull request workflow отсутствует — изменения коммитятся напрямую в main.
- Каждый коммит, затрагивающий `wiki/`, также должен обновить `wiki/index.md` и дополнить `wiki/log.md`.

## Дисциплина лога (`wiki/log.md`)

Журнал операций — **только дополнение (append-only)**. Никогда не редактируйте и не удаляйте старые записи. Формат записи:

```
## [ГГГГ-ММ-ДД] <операция> | <название>
```

Где `<операция>` — одно из: `ingest`, `query`, `lint`, `create`, `update`.

## Инструкции для агента

- **Никогда не изменять файлы в `raw/`** — только читать из этой директории.
- **Никогда не удалять контент из `wiki/`** без одобрения пользователя (исключение: исправление битых ссылок).
- **Предпочитать много маленьких связанных страниц** одной большой.
- При инджесте источника создавать или обновлять **все затронутые страницы** (5–15 страниц на источник — норма).
- Всегда сначала искать информацию в существующей вики, прежде чем обращаться к внешним инструментам — точка входа `wiki/index.md`.
